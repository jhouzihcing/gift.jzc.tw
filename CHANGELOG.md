## [2.11.0] - 2026-04-16
### 🔄 同步引擎完整重構 (Sync Engine Rewrite)

根本重構同步系統，徹底解決「同步中…已同步…同步中…」無限循環但實際未同步至雲端的問題。

**核心問題（已修復）：**
- **舊版樂觀寫入（Flash Sync）的 etag 機制**在 412 衝突後的恢復路徑存在新的錯誤，導致資料無法寫入；同時 `dbRef` 記憶體快取與 Drive 實際狀態產生落差。
- **三條並行同步路徑互相競爭**（即時觸發、30 秒背景輪詢、衝突重試），各自持有獨立計時器，製造無法收斂的循環。
- **`initDrive()` 在每次 token 刷新時重新執行**，產生第四條並行 Drive 寫入，進一步造成狀態混亂。

**新架構（響應式同步引擎）：**
- **讀先寫後（Read-first）**：每次同步前必定先從 Drive 讀取最新狀態，合併本地變更後再寫入。移除 etag，徹底告別 412 衝突問題。
- **響應式觸發**：透過 Zustand 訂閱 `useCardStore` 中的 `cards`，偵測到任何卡片變為 `isSynced: false`（掃描新增、移至垃圾桶、還原）時立即自動入隊同步，無需任何手動觸發。
- **單一佇列 + 互斥鎖**：`queueRef` 為唯一的同步佇列，`busyRef` 防止並行寫入；佇列中的卡片去重，保證每張卡片只處理一次。
- **指數退避**：失敗後依序等待 2s → 4s → 8s → 16s → 32s → 60s（上限）再重試，避免短時間內爆量重試。
- **60 秒安全網**：作為最後防線，定期掃描所有未同步卡片並補傳，確保任何邊緣情況都不漏網。

## [2.10.0] - 2026-04-16
### 🐛 同步卡死修復 (Sync Deadlock Fix)
- **修復掃描後無法立即同步**：`scan/page.tsx` 原本自行呼叫 `useDriveSync()` 建立第二個 hook 實例，導致其 `fileIdRef` 永遠為 null，`syncImmediately` 直接 return 而不進行任何同步。現改為從 Zustand 全域 store 取得 `syncCard`，由 `AuthProvider` 的唯一 hook 實例處理所有同步。
- **修復「永久同步中」**：`flushPending()` 的 `catch` 區塊在同步失敗時從未呼叫 `setSyncStatus(false, ...)`，導致 `isSyncing` 永遠停在 `true`，UI 一直顯示「同步中」。現在錯誤發生時會正確重置狀態，再由重試機制在下次嘗試時重新設為 `true`。
